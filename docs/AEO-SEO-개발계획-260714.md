# AEO / SEO 개발 계획 (2026-07-14)

> 대상: `deepingsource.io` (Next.js App Router · Velite/Keystatic CMS · en/ko/jp 3개국어)
> 목적: 받은 AEO·SEO·기술 피드백을 **현재 코드베이스 기준으로 재평가**하고, 실제로 남은 작업만 우선순위화하여 개발 계획으로 정리.
> 카피 방향: **감성 톤 유지 + 키워드 절충(하이브리드)** — 히어로 감성 카피는 살리되 서브헤드·메타·H2에 타깃 키워드를 보강.

## 0. 핵심 요약 (TL;DR)

받은 피드백의 상당수는 **이미 구현되어 있다.** 글로서리(사전형 정의) ~20개 + `DefinedTerm` 스키마, `Organization`/`WebSite`/`SoftwareApplication`/`Service`/`Article` JSON-LD 헬퍼, 메타데이터·사이트맵 양쪽의 hreflang, 키워드 풍부한 alt 텍스트, 사일로 URL 구조가 모두 존재한다.

따라서 이 계획은 "처음부터 만들기"가 아니라 **남은 갭 5가지를 메우는 것**에 집중한다.

| 우선순위 | 작업 | 임팩트 | 공수 |
|---|---|---|---|
| **P0** | FAQ 페이지에 `FAQPage` JSON-LD 실제 출력 | 높음(AEO 인용률) | 낮음 |
| **P0** | 홈 H1 하이브리드 카피(감성+키워드) + Title 재정비 | 높음(SEO 진입) | 낮음 |
| **P0** | 메타 Title 길이 정리(60자 초과 truncation 해소) | 높음(SERP CTR) | 낮음 |
| **P1** | 제품 현장 사례 → Problem · Solution · Result 구조화 | 높음(B2B 인용) | 중간 |
| **P1** | 메인 키워드 확정("Anonymized Spatial AI") 후 전진 배치 통일 | 중간 | 중간 |
| **P1** | 제품 네이밍 규칙 통일(saai X / store X 혼용 해소) | 높음(명료성) | 중간 |
| **P1** | "그래서?" 다리 없는 추상 카피에 혜택 문장 보강 | 높음(설득·전환) | 중간 |
| **P2** | 본문 내부링크 앵커 텍스트 키워드화 | 중간 | 낮음 |

| **P2** | EN·KO CTA 톤 정합(영문 CTA 명료화) | 중간 | 낮음 |
| **P2** | 전문용어(SAAI·MTMC·SEAL) 최초 등장 시 인라인 글로스 | 중간(AEO) | 낮음 |

## 1. 피드백 항목별 갭 분석 (현황 대조)

범례: ✅ 구현됨 · 🟡 부분 구현 · ⬜ 미구현

### 1-A. AEO 관점

| 피드백 항목 | 현황 | 근거 (코드) |
|---|---|---|
| 핵심 기업 정의(Entity) | ✅ | `layout.tsx` — Organization `description`에 "익명화 공간 AI로 모든 오프라인 공간을…" 정의문 존재 |
| 신뢰도 지표(특허·NVIDIA·파트너) | 🟡 | Organization에 `memberOf: NVIDIA Inception` ✅ / "103 patents", 파트너 수는 스키마 미반영 → 보강 여지 |
| 핵심 용어 사전적 정의 블록 | ✅ | `content` 글로서리 ~20개(store-heatmap, dwell-time, anonymized-cctv, mtmc 계열 등) + `definedTerm()`/`definedTermSet()` 스키마 |
| 용어 정의에 `DefinedTerm` 스키마 | ✅ | `structured-data.tsx` + `GlossaryDetailView.tsx` |
| 사용 사례 Problem→Solution→Result 구조화 | ⬜ | 제품 페이지 설명이 서술형(예: store-insight `description`은 카피 문장). 정량 결과(+10%)가 구조화 안 됨 |
| **FAQ Q&A → `FAQPage` 스키마** | 🟡 → ⬜ | `faqPage()` 헬퍼는 **존재하나** `FaqView`/faq 페이지에서 **호출 흔적 없음** → 스키마 미출력 |

### 1-B. SEO 관점

| 피드백 항목 | 현황 | 근거 |
|---|---|---|
| 풍부한 alt 텍스트 | ✅ | 이미지 alt에 키워드 포함(피드백에서도 강점으로 인정) |
| 사일로 URL 구조 | ✅ | `/products/*`, `/technology/*`, `/solutions/*`, `/resources/*` |
| 메인 H1에 타깃 키워드 | ⬜ | `HeroSection.tsx` H1 = 감성 카피("Perfect your space." 계열), 타깃 키워드 없음 |

| 키워드 일관성(SAAI vs Spatial AI) | ⬜ | 문서 전반 혼용 → 메인 키워드 미확정 |

## 2. 우선순위별 개발 계획

### 🔴 P0-1. FAQ 페이지 `FAQPage` JSON-LD 출력 (AEO 최고 ROI)

**문제:** `faqPage()` 헬퍼가 이미 있는데 FAQ 페이지에서 미사용 → AI 검색엔진이 Q&A로 인식할 근거 부재.

**작업**
- `src/components/corporate/views/FaqView.tsx`(또는 `src/app/resources/faq/page.tsx`)에서 FAQ 데이터 배열을 `faqPage()`에 전달하고 `<JsonLd data={…} />` 렌더.
- FAQ 항목이 컴포넌트 하드코딩이면 `{ question, answer }[]` 배열로 분리 → 화면 렌더와 스키마가 동일 소스 사용(중복 방지).
- en/ko/jp 각 로케일 FAQ에 동일 적용.

**검증:** 빌드 후 `out/resources/faq/index.html`에 `"@type":"FAQPage"` 존재 확인 → Google Rich Results Test 통과.

**공수:** 0.5일 · **임팩트:** 높음

---

### 🔴 P0-2. 홈 H1 하이브리드 카피 (정밀 진단)

**정밀 진단(코드 확인 결과):** `CorporateHero.tsx`의 실제 마크업은 이미 하이브리드에 가깝다.
- **H1** = `perfectSpace.your[locale]` → "Perfect your space." / "당신의 공간을, 완벽하게" (감성)
- **위쪽 eyebrow `<p>`** = "ANONYMIZED SPATIAL AI · SAAI" (키워드 존재)
- **heroSub** = "…Anonymized Spatial AI that keeps no footage." (키워드 존재)

즉 키워드는 **eyebrow와 sub에는 있으나 heading 위계(H1/H2)에는 없다.** 검색엔진·AI는 heading 위계에 가중치를 둔다.

**방향(하이브리드):** 감성 H1은 그대로 두고, 키워드를 **heading 위계로 승격**한다.
- eyebrow `<p>` "ANONYMIZED SPATIAL AI · SAAI"를 `<p>` 대신 시각적으로 동일하되 의미상 상위인 배치로 조정하거나, heroSub 첫 문장을 **H2**로 승격.
  예) H1 "Perfect your space." + H2 "Anonymized Spatial AI for offline retail — 얼굴 없이, CCTV 위에서."
- H1은 페이지당 1개 유지, 승격 문구는 H2.

**검증:** 렌더된 heading 요소(h1·h2)에 "Anonymized Spatial AI" 노출, Lighthouse SEO ≥ 95 유지.

**공수:** 0.5일 · **임팩트:** 높음

---

### 🔴 P0-3. 메타 Title 길이 정리 (SERP truncation)

**문제:** 일부 제품 Title이 60자를 크게 초과해 검색결과에서 잘린다.
- 예: `saai agent — store agent | DEEPINGSOURCE — Store AI that proposes your next move` (약 80자)
- `saai insight — store insight | DEEPINGSOURCE — Spatial analytics that read your store`

**작업**
- 패턴을 `제품명 | DEEPINGSOURCE — 핵심 혜택(키워드 앞)` 55~60자로 통일.
  예) `store agent — Spatial AI for your next move | DEEPINGSOURCE`
- 뒤쪽 설명 문구는 `meta description`으로 이동.

**검증:** 전 페이지 `<title>` ≤ 60자, 타깃 키워드가 앞쪽 절반에 위치.

**공수:** 0.5일 · **임팩트:** 높음

---

### 🟠 P1-1. 제품 현장 사례 → Problem · Solution · Result 구조화

**문제:** 사례가 서술형이라 AI가 B2B 솔루션 추천 시 인용하기 어려움.

**작업**
- 각 제품/솔루션 페이지의 사례 블록을 3단 구조로 재작성:
  - **Problem**: "편의점 야간 결품으로 주말 매출 손실"
  - **Solution**: "store insight의 결품 감지 + 동선 분석"
  - **Result**: "리오더 리드타임 단축 → 매출 +10%"
- 마크업: 각 결과를 `<dl>`/명시적 라벨로 → 정량 수치를 텍스트로 노출(이미지 안에 가두지 않기).
- 여력 시 사례에 `Article` 또는 케이스스터디 스키마 부여(`resources/case-studies` 이미 존재).

**검증:** 사례 3건 이상이 Problem/Solution/Result 라벨 + 정량 수치 텍스트 포함.

**공수:** 2~3일 · **임팩트:** 높음

---

### 🟠 P1-2. 메인 키워드 확정 및 전진 배치 통일

**결정 필요:** 트래픽 허브 키워드 = **"Anonymized Spatial AI(익명화 공간 AI)"** 권장(홈 Title·Organization 정의문과 이미 정합).

**작업**
- SAAI는 제품 브랜드명, Spatial AI는 기술/카테고리 키워드로 **역할 분리** 규칙 문서화.
- 제품 설명 도입부·H2·메타 첫머리에 카테고리 키워드 전진 배치.
- `keywords` 배열(메타)·글로서리 상호링크에 일관 반영.

**검증:** 주요 5개 페이지 Title/H1-H2/첫 문단에 카테고리 키워드 1회 이상 노출, 표기 불일치 0건.

**공수:** 1~2일 · **임팩트:** 중간

### 🟡 P2-1. 본문 내부링크 앵커 텍스트 키워드화

**작업**
- 버튼형 CTA("Find your fit", "See the live demo")는 유지하되, **본문 내부**에 키워드 앵커 추가.
  예) "…자세히는 [MTMC 다중 카메라 추적](/technology/spatial-ai)에서" / "[결품 감지](/glossary/stockout-detection)"
- `ProductCard`/`CtaBand`의 제너릭 "Learn more"는 문맥 키워드로 치환 검토.
- 글로서리 용어를 본문에서 내부링크로 자연 연결(토픽 클러스터 강화).

**검증:** 주요 페이지당 키워드 포함 내부링크 ≥ 2개.

**공수:** 1일 · **임팩트:** 중간

### 🟡 P2-2. 스키마 커버리지 감사 + 신뢰지표 보강

**작업**
- 빌드 산출물(`out/**/index.html`) 스캔으로 각 제품 페이지 `SoftwareApplication`, 솔루션 페이지 `Service`, 블로그 `Article` 실제 출력 여부 확인 → 누락 페이지에 헬퍼 연결.
- Organization 스키마에 신뢰지표 보강: `award`/`hasCredential`(특허 103건), 파트너를 `subjectOf`·설명 텍스트로.
- (선택) `slug` 기반 자동 검증 스크립트를 `scripts/`에 추가해 회귀 방지.

**검증:** 전 페이지 유형별 스키마 1종 이상 출력, Rich Results Test 에러 0.

**공수:** 1일 · **임팩트:** 중간

## 2-B. 전체 카피 재검수 (7-스윕 진단)

David Ogilvy 위계 + 7-스윕(명료성 → 보이스 → 그래서? → 증명 → 구체성 → 감정 → 위험제로)으로 실제 렌더 카피를 점검했다. **브랜드 보이스 자체는 강하다** — `brand-canon.ts` SOT, Voice_v2.5, 두 마스터 카피(회사 "모든 공간을, 완벽하게" · 점주 "우리 매장이 대표 매장처럼")로 잘 관리되고 있다. 다만 **시적·추상적 카피가 검색·전환 지점에서 구체 혜택과 키워드를 놓치는** 패턴이 반복된다.

### 진단 요약 (스윕별)

| 스윕 | 발견 | 심각도 |
|---|---|---|
| 1 명료성 | 제품 네이밍 이중화(`saai insight`/`store insight`, `store count`는 saai 접두 없음, `saai.store`) → 콜드 방문자·AI 혼선 | 높음 |
| 1 명료성 | H1 "Perfect your space."는 콜드 방문자에게 무엇을 파는지 전달 안 됨(키워드는 eyebrow에만) | 높음 |
| 2 보이스 | EN CTA는 시적("Map your fastest path"), KO CTA는 직설("우리 매장 진단받기") → 로케일 간 행동 명료성 불일치 | 중간 |
| 3 그래서? | "Beyond data, to decisions." / "Beyond seeing — AI that operates space." — 혜택 다리 없음 | 높음 |
| 4 증명 | 신뢰지표(특허 103·NVIDIA·파트너)는 강함. 단 현장 사례 결과(+10%)가 "illustrative"로만 표기, 귀속(attribution) 약함 | 중간 |
| 5 구체성 | "Perfect every space" 류 추상 vs "카메라 한 대로, 매일, 무인" 류 구체 — 편차 큼 | 중간 |
| 6 감정 | 점주 보이스("우리 매장이 대표 매장처럼")는 감정적으로 강함 — 유지 | (강점) |
| 7 위험제로 | "영상은 남기지 않는" · "무료 상담 · 1–2일 내 회신" 등 안심 신호 양호 — 유지 | (강점) |

### 개선 작업 (before → after 예시)

아래는 **방향 예시**다. 실제 확정은 브랜드 SOT(`SAAI_AI_Handoff.md`) 사인오프 필요 — 마스터 카피는 건드리지 않고 **받치는 문장·혜택 다리·CTA·메타**만 손본다.

**(a) 제품 네이밍 규칙 통일 [P1]**
- 현재: `saai insight — store insight`, `saai agent`, `saai care`, `store count`(접두 없음), `saai.store`(B2C).
- 방향: 규칙 1개 확정 후 전 페이지·메타·JSON-LD `alternateName`에 통일.
  - 권장: 엔터프라이즈 4종은 `store {insight·care·agent·count}`를 정식 명칭으로, `saai {…}`는 `alternateName`(부제)로만. B2C는 `saai.store`로 분리 유지.
- 영향 파일: `site-content.json`, 각 `products/*/page.tsx` title, `structured-data.tsx` softwareApplication `alternateName`.

**(b) "그래서?" 다리 추가 [P1]**

| Before | After (방향) |
|---|---|
| store agent: "Beyond data, to decisions." | "데이터를 결정으로 — 오늘 무엇을·얼마나·어디에 둘지 한 줄로." |
| signature: "Beyond seeing — AI that operates space." | (마스터는 유지) 받침 문장 추가: "보기만 하는 CCTV를, 발주·배치까지 실행하는 AI로." |
| store care: "Peace of mind, by the hour." | "시간 단위 안심 — 도난·화재·이상을 대신 지켜보고 바로 알립니다." |

**(c) EN CTA 명료화 [P2]**
- "Map your fastest path" → "Get a free store diagnosis"(KO "우리 매장 진단받기"와 정합).
- 시적 톤은 헤드라인에서 살리고, **버튼(행동 지점)은 직설**로 — 로케일 간 일관.

**(d) 전문용어 인라인 글로스 [P2]**
- SAAI·MTMC·SEAL·anonymizer·Spatial AI 최초 등장 시 한 줄 정의 + 글로서리 내부링크.
  예) "MTMC(다중 카메라 간 동일 인물 연속 추적)로…" → `/glossary`·`/technology/spatial-ai` 링크.
- AEO상 정의문은 AI가 그대로 인용 → §1-A 사전 정의 자산과 시너지.

**(e) 증명 강화 [P1-1과 연계]**
- CaseBand "illustrative" 사례를 가능한 범위에서 실명·수치 사례로 승격(§P1-1 Problem·Solution·Result 구조와 동일 작업).

### 유지할 카피 자산 (건드리지 말 것)
- 두 마스터 카피, 카테고리 정의문("이미 달린 CCTV 위에서, 누구가 아니라 무엇을 어떻게…"), 프라이버시 안심 문장("영상은 남기지 않는"), 점주 보이스, 신뢰지표 배지. 이들은 브랜드 SOT 기반이며 AEO 엔티티 정의로도 이미 기능하고 있다.

## 3. 실행 순서 제안 (2주 스프린트)

- **주차 1**: P0-1(FAQ 스키마) → P0-2(H1 승격) → P0-3(Title 길이) → P1-2(키워드 확정)·네이밍 규칙 문서화
- **주차 2**: P1-1(사례 구조화=증명 강화) → 카피 "그래서?" 다리 보강 → P2-1(앵커)·EN CTA 정합 → P2-2(스키마 감사)·전문용어 글로스

> 카피 변경은 대부분 `site-content.json`·`brand-canon.ts`·`products/*/page.tsx`·뷰 컴포넌트의 dict에 집중된다. 마스터 카피는 브랜드 SOT 사인오프 전까지 동결.

각 항목 완료 시 공통 검증: `npm run build` → `out/` 정적 산출물에서 스키마/hreflang grep → Google Rich Results Test & Lighthouse SEO.
