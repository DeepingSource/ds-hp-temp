# DeepingSource 홈페이지 전체 페이지 감사 및 개선안

- **감사 대상:** https://ds-hp-temp.vercel.app/ (임시/프리뷰 배포, EN 로케일 기준)
- **감사일:** 2026-07-14
- **방법:** 전 페이지 순차 브라우징(스크린샷 + 렌더링된 텍스트 추출) — 홈, 제품 5종, 기술 4종, 솔루션 허브+업종 4종, 가격 2종, 리소스 5종, 회사 5종, 기타(문의/엔터프라이즈/데모/헬프/법무 등)
- **범위 메모:** 개별 솔루션 상세(24종), 블로그 글 250여 건, 용어 19건은 허브·목록 페이지와 대표 샘플로 검토했고 전수 클릭은 하지 않았습니다. ko/jp 로케일은 이번 1차 감사에서는 EN 기준으로 진행했고, 발견된 언어 혼용 이슈 자체가 로케일 간 대조가 필요하다는 근거이기도 합니다.

이 문서는 기존 `docs/` 산출물(7/1 평가안, 7/14 피드백 반영 계획 등)을 참고하지 않고 라이브 사이트만 다시 훑어서 작성한 **독립적인 신규 감사**입니다. 겹치는 항목도 있지만, 실제 화면에서 방금 확인한 근거(스크린샷·본문 텍스트)를 기준으로 우선순위를 다시 매겼습니다.

## 0. 총평

콘텐츠 밀도와 카피 완성도는 전반적으로 높습니다. 특히 제품 상세(store insight/agent/count), 회사 소개, 투자자 페이지는 구체적 수치와 사례로 뒷받침되어 신뢰도가 좋습니다. 반면 **사이트 전체를 관통하는 5가지 구조적 문제**가 여러 페이지에서 반복적으로 발견됐고, 이것이 개별 페이지의 완성도보다 방문자 신뢰에 더 큰 영향을 줄 것으로 보입니다.

| 영역 | 평가 | 핵심 이슈 |
|---|---|---|
| 제품 브랜딩/네이밍 | 미흡 | "saai insight" vs "store insight" 등 동일 제품에 두 이름이 병기·혼용 |
| 다국어(EN 페이지) | 미흡 | 한국어 원문이 영문 페이지 곳곳에 그대로 노출 (파트너명, 캡션, 페이지 전체 등) |
| 데이터·증거 신뢰성 | 보통 | "Measured, not promised"라는 페이지 제목과 달리 가상 인터뷰·가상 수치가 실사례처럼 배치 |
| 정보 노출 범위 | 위험 | 내부용 목업 리뷰 페이지·CMS 편집 가이드가 인증 없이 공개 URL로 노출 |
| 콘텐츠 완성도 | 보통 | Docs 허브의 90% 이상이 "coming soon", 보도자료 최신 항목이 1년 이상 전 |

## 1. 사이트 전역 이슈 (여러 페이지에 걸친 구조적 문제)

### [P0] 내부 전용 페이지가 인증 없이 공개 URL로 노출

- **`/demo`**: 페이지 제목이 `목업 데모 리뷰 (내부용)` — "내부용"이라고 스스로 명시한 디자인/애니메이션 검토용 목업 페이지가 로그인 없이 공개 도메인에서 그대로 열립니다. 28종 목업, 로케일 토글, "애니메이션 정지" 버튼 등 명백히 내부 QA 도구입니다.
- **`/help`**: 페이지 제목 `편집 가이드`. Keystatic CMS 사용법(깃허브 로그인, 문서-페이지 매핑표, 커밋/배포 흐름 등 내부 운영 절차)을 담은 편집자용 문서가 전부 한국어로 공개 URL에 있습니다.
- **위험:** 검색엔진에 색인되거나 잠재 고객·경쟁사가 우연히 발견하면 "완성되지 않은 내부 문서를 공개 방치"라는 인상을 주고, 편집 프로세스(깃허브 저장소명 `DeepingSource/ds-hp-temp` 등)까지 노출됩니다.
- **권장:** 두 경로 모두 (a) `noindex` + `robots.txt` disallow를 최소한으로 적용하고, (b) 가능하면 인증 게이트(사내 IP 제한, 비밀번호, 또는 별도 스테이징 도메인)를 걸고, (c) 프로덕션 배포 시 라우트 자체를 빌드에서 제외하는 방안을 검토하세요.

### [P0] 동일 제품에 두 개 이상의 이름이 병기·혼용됨

제품 허브(`/products`)의 순환 다이어그램 카드에는 "saai insight / store insight", "saai care / store care", "saai agent / store agent"가 **한 카드 안에 위아래로 같이** 적혀 있습니다. 반면:
- 브레드크럼과 배지는 "saai insight · store insight" 식으로 병기
- 본문 카피는 거의 전부 "store insight/care/agent"만 사용
- `/technology` 하단 "Products powered by this technology"는 "store insight / store agent / store care"만 사용 (saai 접두어 없음)
- 홈 "Three stores, one flow" 섹션도 "store insight/care/agent"만 사용
- 반면 홈 상단 제품 캐러셀 탭과 푸터는 "store count / saai insight / saai care / saai agent"로 또 다르게 표기
- 블로그에는 더 오래된 이름 "PLUS INSIGHT"까지 등장

방문자 입장에서는 설명 없이 같은 제품이 여러 이름으로 불리는 것처럼 보입니다. 리브랜딩(saai → store) 과정의 잔재로 추정되며, **정식 명칭을 하나로 확정하고 전 페이지에서 일괄 치환**하는 작업이 필요합니다. 부득이 두 이름을 병기해야 한다면 최초 1회 "이전 명칭: X" 식으로 명시하는 게 낫습니다.

추가로 "saai"라는 단어 자체가 3가지 다른 의미로 쓰입니다: ① 브랜드/기술 약칭(Spatial·Anonymized·Agentic·Intelligence), ② 제품 접두어(saai insight 등), ③ 별도 B2C 앱 "saai.store"의 이름. `/products/saai` 페이지는 실제로는 saai.store라는 완전히 다른 소비자용 앱을 소개하는 페이지라 더 헷갈립니다. 이 3중 의미 중첩을 정리하는 것이 근본 해결책입니다.

### [P1] 가격·통화 표기가 페이지마다 다름

같은 store care 요금(월 14,900)이 세 가지 다른 형식으로 표기됩니다:

- `/products/store-care`: **₩14,900/month**
- `/pricing`: **14,900원~/mo**
- `/pricing/simulator`: **14,900 KRW~/mo**

가격은 방문자가 가장 꼼꼼히 확인하는 정보인 만큼, 통화 기호·통화 코드 표기를 하나로 통일해야 합니다(국제 방문자를 고려하면 "KRW" 또는 "₩" 중 하나로 통일 권장, "원"은 EN 페이지에서 제거). 또한 `/pricing/simulator`의 매장 규모 단위 **"평(pyeong)"**은 국제 방문자에게 생소하므로 ㎡/sqft 환산값을 병기하는 것을 권장합니다.

### [P1] "Measured, not promised"라는 페이지 제목과 실제 콘텐츠의 괴리

`/resources/case-studies`는 "Measured, not promised"(측정된 것이지 약속이 아니다)라는 강한 신뢰 메시지로 시작하고, 상단 고지문에는 "Measured 수치는 2025년 12월~2026년 1월 사이 53개 매장 실측"이라고 명시합니다. 그런데 확인한 5번째 케이스(전시·박물관)는:

- 핵심 지표 4개 전부 "Illustrative"(가상) 라벨
- 고객 인용문 2개가 각각 "— Facility Operations Director, Venue D **(illustrative)**"로, **가상의 직함과 가상의 시설명으로 만든 가짜 인터뷰**가 실제 후기처럼 배치됨

"측정된 것이지 약속이 아니다"라는 강력한 신뢰 문구 바로 아래에 가상 인터뷰가 실제처럼 놓여 있는 것은 페이지 제목이 주는 신뢰와 정면으로 배치됩니다. 실측 사례와 가상 시나리오는 시각적으로 명확히 분리(예: 배경색, 큰 배지, "가상 시나리오" 섹션 타이틀)하거나, 실측이 준비되지 않은 케이스는 별도로 "예정" 처리하는 것을 권장합니다. 홈페이지의 "Field scenarios"(문 앞 편의점 등)와 솔루션 허브의 익명 오너 인용문도 같은 패턴(회사명 없는 일반 화자 + Illustrative 라벨)이라 전반적으로 "실제 고객 사례 vs 가상 시나리오" 구분 규칙을 사이트 전체에 명문화할 필요가 있습니다.

### [P1] 업종 허브 4개 중 3개가 동일한 H1 카피 사용

`/solutions/retail`, `/solutions/food-beverage`, `/solutions/drug` 세 페이지가 토씨 하나 틀리지 않고 동일한 헤드라인 "Different formats, one store"를 씁니다(`/solutions/large-space`만 "Every space, like one."로 다름). 업종별 차별화된 메시지가 목표라면 최소 헤드라인만이라도 업종 고유 pain point로 새로 써야 합니다.

### [P2] 헤드라인 등장 애니메이션이 첫 로드 시 텍스트를 깨진 것처럼 보이게 함

거의 모든 페이지(홈, 제품 5종, 기술 4종 등)에서 스크린샷을 찍는 순간 H1이 줄 단위로 겹치거나 잘려 보이는 현상을 반복적으로 확인했습니다(예: "Erase _ Keep _"처럼 단어 일부만 보임). 애니메이션 자체가 끝나면 정상 렌더링되지만, 첫 진입 시 1~2초간 헤드라인이 읽기 어려운 상태로 노출되는 것은 최초 인상에 부정적입니다. 등장 애니메이션의 지속시간을 줄이거나, 최소한 레이아웃 시프트 없이 텍스트가 처음부터 읽을 수 있는 상태로 페이드인되도록 조정을 권장합니다.

### [P2] 인물/사실 정보 불일치

- CBO 이름이 회사소개(`/company/about`)에서는 **"Sang-ho Geum"**, 보도자료(`/company/news`)에서는 **"Sang-ho Kim"**으로 다르게 표기됩니다. 실존 인물의 이름 표기 오류는 신뢰도에 직접 영향을 주므로 최우선 교정 대상입니다.
- `/technology/models` 모델 카탈로그 표는 "13 models"라고 명시하지만 실제 표 행은 13개, 하단 인터랙티브 칩 목록은 18개(cam-calibrate, floor-project, change-detect, synth-frame 등 표에 없는 항목 포함)로 서로 다른 두 목록이 병존합니다.
- `/technology/spatial-ai`의 "References" 섹션은 "CES 전시, KDDI 협업, NVIDIA 생태계에서 발표·인용되었다"고 언급하지만 실제 링크나 자료가 없어 확인이 불가능합니다. 근거 자료가 있다면 링크를, 없다면 문구를 완화하는 것을 권장합니다.

### [P2] Docs 허브가 사실상 비어 있음

`/resources/docs`에 나열된 16개 문서 항목 중 15개가 "Doc coming soon"입니다. 실제 콘텐츠가 있는 것은 store insight 사용자 매뉴얼 1건뿐입니다. 리소스 허브에서는 "product docs"를 주요 콘텐츠 축으로 안내하는데 정작 열어보면 대부분 비어 있어 첫인상이 실망스러울 수 있습니다. 문서화 로드맵이 있다면 "coming soon" 대신 예상 시기를 명시하거나, 완성된 것만 노출하고 전체 개수를 실제 완성분 기준으로 표시하는 것을 권장합니다.

### [P2] Press & Media가 오래됨

`/company/news`에 등록된 보도자료 중 가장 최근 항목이 2025년 6월(감사 시점 기준 약 13개월 전)입니다. 실제로 그 사이 활동이 없었다면 괜찮지만, 투자·글로벌 확장 등 활발히 진행 중인 회사로 소개하는 것과 비교하면 보도자료 갱신이 뒤처진 인상을 줍니다.

### [P2] 도메인/제품 구조가 복잡함

사이트 안에서 언급되는 도메인만 최소 5개입니다: `deepingsource.io`(본사), `saai.store`(B2C 앱), `storecare.ai`(B2C 앱), `app.storeinsight.ai`(대시보드 예시), `app.saai.ai`(엔터프라이즈 대시보드 예시, `/enterprise`에서만 등장). 각 페이지에서 "왜 다른 도메인으로 가는지"는 대체로 잘 설명되어 있지만, 전체 구조를 한눈에 보여주는 사이트맵/다이어그램이 없어 처음 방문자는 "이게 다 같은 회사 맞나?" 싶을 수 있습니다. `/products` 허브의 "FOR OWNERS · B2C (SEPARATE SITES)" 섹션을 확장해 전체 제품·도메인 관계도를 시각적으로 보여주는 것을 권장합니다.

## 2. 섹션별 상세 발견 사항

### 2.1 Home (`/`)

- **좋은 점:** "382명 입장 → 65명 결제, −317의 행동이 사라진다"는 퍼널 시각화가 여전히 강력합니다. Observe→Analyze→Suggest→Learn 서사, SEAL 원본/익명화 슬라이더, MTMC 데모 모두 완성도가 높습니다.
- **[P1]** "Three stores, one flow" 섹션은 store insight/care/agent 3개만 보여주고 store count가 빠집니다. 반면 바로 위 섹션은 "One operating loop, **four** products"라며 4개를 강조합니다. 왜 이 섹션만 3개인지 설명이 없어 숫자가 안 맞아 보입니다.
- **[P2]** 제품 캐러셀이 자동 재생되는 동안 탭 하이라이트(파란색)와 아래 표시되는 콘텐츠가 한 프레임 어긋나는 순간이 있습니다(예: "store count" 탭이 하이라이트된 상태에서 콘텐츠는 "saai agent"를 보여줌). 전환 타이밍을 탭 하이라이트와 콘텐츠 교체가 동시에 일어나도록 맞추는 것을 권장합니다.
- **[P2]** 푸터 위 카피 "The force that remade online — now for every space where people work and live."는 "online"이 무엇을 가리키는지(인터넷 산업 전반? 이커머스?) 불분명해서 곱씹어야 이해되는 문장입니다. 더 직접적인 문장으로 바꾸는 것을 권장합니다.

### 2.2 Products (`/products` + 5개 서브페이지)

- **[P0]** (전역 이슈 참조) 순환 다이어그램에 "saai X / store X" 두 이름이 나란히 표시되는 것이 이 섹션에서 가장 두드러집니다.
- **좋은 점:** `/products/store-insight`는 방법론(기간 비교, 피어 조정, 소스 구분)까지 설명하는 등 콘텐츠 깊이가 매우 좋습니다. `/products/store-count`의 "수동 계수 대비 정확도 95~96%" 같은 구체적 검증 수치도 신뢰를 높입니다.
- **[P1]** `/products/store-care`는 상단에 "From ₩14,900/month"라고 요약하고, 하단 "Care pricing"에는 "First month free · ₩34/hour · ₩14,900/month"가 다시 나옵니다. 시간당/월정액이 함께 제시되는데 언제 어느 쪽이 적용되는지 설명이 없어 혼란을 줄 수 있습니다.
- **[P2]** `/products/saai`(saai.store 소개 페이지) 가격표 안에 "Premium · sister products"로 store care/store insight가 다시 등장합니다 — B2C 앱 가격표 안에 B2B 제품이 교차 판매되는 구조 자체는 흥미롭지만, 페이지 제목(`saai.store`)과 URL(`/products/saai`)이 실제 콘텐츠와 매칭되는지 재검토가 필요합니다.

### 2.3 Technology (`/technology` + 4개 서브페이지)

- **좋은 점:** Anonymizer, SEAL, Spatial AI 페이지 모두 4단계 메커니즘 다이어그램, FAQ, 스펙 표가 있어 기술 신뢰도가 높습니다. "Full vs Partial anonymization" 비교표도 설득력 있습니다.

## 4. 다음 단계 제안

1. **표기 정본(single source of truth) 확정** — 제품명(saai X vs store X), 통화 표기, 인물 이름 등 "정답"을 하나의 브랜드 표기 가이드 문서로 먼저 확정한 뒤 전 페이지 일괄 치환. (이 작업 없이 개별 페이지를 고치면 같은 불일치가 다시 생깁니다.)
2. **공개 위험 요소 즉시 조치** — `/demo`, `/help`는 콘텐츠 수정보다 접근 제어가 우선입니다.
3. **i18n 커버리지 스캔** — 소스 문자열과 렌더링 결과를 비교하는 스크립트로 "한국어가 남은 위치"를 전수 조사하면 이번 감사에서 발견한 것 외의 누락도 찾을 수 있습니다.
4. **"실측 vs 가상" 표기 규칙 수립** — Illustrative 라벨을 다는 것에서 한 걸음 더 나아가, 시각적으로도 실측 사례와 구분되도록 컴포넌트 레벨에서 규칙화.
