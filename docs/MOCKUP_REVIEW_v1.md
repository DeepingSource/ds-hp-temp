## 0. 총평

기반은 이미 상위권이다. 디바이스 프레임 통일, 3개 로케일 i18n, reduced-motion 가드, 스크롤 게이트 루프(`useMockupLoop`), SSR-안정 결정적 지터, TapIndicator로 시뮬레이션된 탭 — 구조적 완성도는 손댈 곳이 거의 없다.

남은 격차는 구조가 아니라 **"한 매장의 하루"라는 일관된 세계관**이다. 지금은 목업마다 숫자·시간·도메인이 따로 놀아서, 자세히 보는 방문자(특히 본부 실무자)에게 "잘 만든 무대 세트"라는 인상을 준다. 개선의 절반은 코드가 아니라 데이터 정합성 작업이다.

## 1. 콘텐츠 현실성 (가장 큰 격차)

### 1-1. 🔴 같은 강남역점인데 일매출이 3개

| 목업 | 값 | 출처 |
|---|---|---|
| ActionCardMockup (StoreAgent) | 예상 매출 **₩847,000** | `storeagent.ts` todayStats |
| StoreInsightDesktopMockup | 예상 일매출 **124.5만** | `storeinsight.ts` desktopMetrics |
| MultiStoreDashboardMockup | 일 매출 **142만** | `enterprise.ts` stores[0] |

같은 페이지 흐름에서 세 목업을 연달아 보면 바로 들킨다. 방문자·체류시간·전환율도 동일하게 어긋난다.

**개선안**: `src/data/mockup-scenarios/canonical.ts` 한 파일에 "강남역점의 오늘" 캐노니컬 데이터(일매출, 방문자, 체류, 전환율, 시간대 곡선, 알림 건수)를 정의하고 세 시나리오 파일이 모두 여기서 파생하도록 리팩터. 이미 박람회 TF의 `가상가맹점데이터시나리오_v0.1.md`가 같은 문제를 풀어놨으니(잠실광적점 G001 등) 그 수치 체계를 웹 목업의 SOT로 승격하면 박람회 데모 ↔ 홈페이지 목업까지 일치한다.

### 1-2. 🔴 시간 정합성 — 미래 시간대 막대가 차 있다

StoreInsight(폰/데스크탑)는 "오늘" 대시보드인데 10시~21시 전 구간 막대가 가득하다. 오후에 보면 미래 매출이 이미 존재하는 셈. 또 데스크탑 차트는 3초마다 **전체 막대가 ±12% 출렁**이는데(`Math.sin` 지터), 실제 시간대별 누적 차트는 과거가 흔들리지 않는다.

**개선안**:
- 현재 시각(`getTimeSlot` 이미 보유) 이후 막대는 점선/저채도 "예측" 스타일로 분리. AI 제품이므로 **예측 구간 표시 자체가 셀링 포인트**가 된다.
- 지터는 "현재 시간" 막대 1개만 실시간 틱업(+카운트업)으로 한정. 과거 막대는 고정.
- 인사이트 문구("14시 매출 +23%")도 현재 시각 기반으로 동적 생성 — `BriefingMockup`은 이미 timeSlot 분기를 하고 있어 패턴 재사용 가능.

### 1-3. 🟡 도메인 어긋남

- 브라우저 URL이 `app.storeinsight.ai`(StoreInsightDesktop) vs `app.saai.ai`(MultiStore)로 불일치. 제품 도메인 체계를 하나로(예: `app.saai.ai/insight/...`).
- `AnonymizationMockup`만 `locale='ko'` 디폴트(나머지는 `'en'`).
- 같은 매장 라인업인데 MultiStore는 강남/홍대/잠실/신촌/건대, 박람회 SOT는 잠실광적점 체계 — 1-1 통합 시 함께 정리.

### 1-4. 🟡 AI다움의 깊이

현재 인사이트 문구는 좋은 편(근거 수치 + 액션 연결). 한 단계 더:

- **신뢰도/근거 메타 추가**: "전주 동시간 대비 · 신뢰도 87%" 같은 작은 메타 라인. 실제 ML 제품 UI의 시그니처.
- **액션의 결과 환류**: ActionCard 승인 후 empty state에 "오늘 AI 제안 5건 승인 → 예상 효과 +₩31,000" 같은 누적 효과 표시. 승인이 '소비'가 아니라 '성과'로 닫힌다.
- ChatMockup 사용자 메시지가 즉시 등장 — 입력창에 글자가 타이핑되고 전송 버튼이 눌린 뒤 버블 등장으로 바꾸면 체감 현실성이 크게 오른다(이미 TapIndicator·whileTap 인프라 보유).

### 1-5. 🟡 AnonymizationMockup — 유일하게 '그림'처럼 보이는 목업

CCTV 장면이 플랫 벡터 일러스트(살구색 원 얼굴)라 다른 목업의 실사 CCTV(`cctvImages`) 대비 결이 깨진다. 비식별화는 회사의 핵심 기술인데 가장 장난감 같은 화면이 됐다.

**개선안**: 기존 `cctvImages` 실사 스틸 위에 (a) 얼굴 영역 좌표에 CSS 모자이크(확대 + blur + clip) 또는 사전 가공한 픽셀화 이미지 레이어, (b) 기존 bounding box/AI 라벨 오버레이를 그대로 재사용. before/after 슬라이더 구조는 유지. 개인정보 이슈는 이미 비식별 처리된 데모 이미지를 쓰면 해소(박람회 데모 런북과 동일 소재 활용 가능).

## 2. 디자인

### 2-1. 🟡 고정 px 타이포의 스케일 문제

목업 내부 텍스트가 `text-[10px]`~`text-base` 고정인데 프레임은 280→400px로 늘어난다. 작은 뷰포트에서 비율이 깨지고 480px 이상에서는 여백이 과해진다.

**개선안**: 프레임 단위 container query(`@container`) 또는 루트 `font-size` + `em` 체계로 목업 내부를 비율 스케일. 일괄 적용이 부담이면 `transform: scale()` 래퍼(레이아웃은 lg 기준 1벌만 유지)가 가장 저렴하다.

### 2-2. 🟡 차트 렌더링 디테일

- `scaleY` 애니메이션은 막대의 `rounded-t-sm`을 함께 찌그러뜨린다(MultiStore, StoreInsight 폰). `height` 트랜지션(데스크탑 방식)으로 통일하거나 `clip-path` 사용.
- 데스크탑 트렌드라인 폴리라인은 막대와 시각 정렬은 맞지만, 막대-라인 이중 인코딩이 같은 데이터라 정보가 중복. 라인은 '전주'로, 막대는 '오늘'로 역할을 분리하면 범례(방문자/전주)와도 맞는다.
- 히트맵이 완전 정적인데 "실시간" 뱃지가 붙어 있음 — 셀 1~2개 opacity를 주기적으로 미세 변화시키는 것만으로 충분.

### 2-3. 🟢 통일 잘 된 것 / 유지

`MOCKUP_SCHEME`·`PRODUCT_THEME`·`MOCKUP_DEVICE` 토큰, 제품별 헤더 보더 컬러 코딩, MockupBadge("예시 화면") 컴플라이언스 표기, aria-hidden 처리. 이 체계는 그대로 두고 위 항목들만 토큰에 흡수.

### 2-4. 🟢 (선택) 다크/라이트 결 정리

AnonymizationMockup만 다크. 의도적(보안 기술 = 다크 콘솔 결)이라면 TechnologyView 계열 목업 전체를 다크로 묶어 "제품=라이트, 기술=다크" 룰을 명문화하면 우연이 규칙이 된다. DESIGN_v2의 4-레이어 시각 차별화와도 맞물린다.

## 3. 애니메이션

### 3-1. 🔴 관람에서 조작으로 — 인터랙티브 전환

현재 모든 목업이 100% 자동 재생이다. 데모의 설득력은 "내가 만져봤다"에서 나온다. 우선순위 높은 3곳:

1. **Anonymization 슬라이더**: sin 왕복(기계적)을 사용자 드래그 가능으로. 유휴 시에만 자동 왕복 재개. 효과 대비 비용이 가장 낮고 "비식별화를 내 손으로 확인했다"는 경험이 강력하다.
2. **호버/터치 시 루프 일시정지**: 모든 목업 공통. 들여다보려는 순간 장면이 넘어가는 게 현재 최대 UX 마찰. `useMockupLoop`에 `pauseOnHover` 옵션 하나로 해결.
3. **StoreInsight 세그먼트 탭**: 5초 자동 순환 + 클릭으로 즉시 전환(클릭 후 타이머 리셋).

### 3-2. 🟡 타이밍 드리프트

StoreCareMockup은 phase 루프(합 9000ms, `useMockupLoop`)와 시나리오 전환(별도 `setInterval` 9000ms)이 독립 타이머라 장시간 노출 시 어긋나 시나리오가 phase 중간에 바뀐다. 시나리오 인덱스를 루프 사이클 수에서 파생(`cycle % list.length`)시켜 단일 타이머로.

### 3-3. 🟡 ActionCard 리듬

10초 안에 5장 전부 순차 승인 → 너무 부지런한 점주처럼 보이고, 패턴이 단조롭다. 개선: 3장 승인 + 1장 "나중에"(좌측으로 다르게 퇴장) + 1장 잔류 후 루프. '거절'이 섞이면 AI가 강요가 아니라 제안이라는 메시지도 함께 전달된다.

### 3-4. 🟢 (선택) 스크롤 연동 스토리텔링

히어로/제품 상단 1곳 정도는 `useScroll` 기반으로 phase를 스크롤 진행에 묶으면(스크럽) 체감 완성도가 크게 오른다. 단 1곳만 — 전 목업에 적용하면 피로해진다.

### 3-5. 🟢 이미 잘 된 것

springBubble/springGentle/springNotif 스프링 프리셋 분리, 타이핑 인디케이터, visibilitychange 루프 리셋(Chat/Push — `useMockupLoop`에도 동일 처리 내장 권장), exit 애니메이션 후 상태 교체(React 18 배칭 회피), reduced-motion 전 목업 대응.
