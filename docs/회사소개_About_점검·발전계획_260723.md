## 0. 이 문서의 목적과 한 문장 결론

About 페이지를 **한 번씩 짚어가며 "우리가 누구인가 → 왜 이들이 오프라인 문제를 풀 수 있는가 → 그래서 지금 무엇을 하면 되는가(CTA)"** 가 하나의 끊김 없는 흐름으로 읽히도록 재설계한다.

**한 문장 진단:** 지금 About은 *재료는 다 좋은데 순서와 시각 언어가 흩어져 있다.* 좋은 카피(brand-canon SOT)와 좋은 데이터(팀·연혁·인증)가 있지만, ① 히어로가 은퇴한 슬로건으로 시작하고 ② 같은 메시지(Vision 2031 / 3층 구조)가 서로 다른 톤의 컴포넌트로 중복 표현되며 ③ 리더십은 팀 페이지보다 빈약한 별도 데이터를 쓰고 있어, "한 흐름"이 아니라 "섹션 모음"으로 읽힌다.

## 2. 서사 흐름 재설계 — "한 흐름"의 뼈대

지금은 섹션이 8개지만 독자 머릿속의 질문 순서와 어긋난다. 아래 **6비트 서사**로 재정렬한다. 각 비트는 "독자가 그 지점에서 던지는 질문"에 답하고, 다음 질문으로 자연스럽게 넘긴다.

| 비트 | 독자의 질문 | 답(섹션) | 전환 장치 |
|---|---|---|---|
| **1. 선언** | "이 회사, 한마디로?" | 히어로 — 온라인은 최적화됐지만 현장은 그대로. 우리는 그 현장을 다시 만든다. **+ 시각적 clue** | ↓ "왜 하필 당신들?" |
| **2. 자격(Why-us)** | "왜 이들이 오프라인을 풀 수 있나?" | ④ 왜냐하면 → **위치 이동**: 히어로 직후로. 익명화 뿌리(2018)·카메라는 사람을 본다는 원류 서사 | ↓ "그래서 뭘 만드나?" |
| **3. 방법(How)** | "어떻게 공간을 다치지 않고 읽나?" | ③-B **3층 구조**(익명화→공간지능→에이전트) — *단독 화이트 섹션으로 승격* | ↓ "그게 지금 어디까지?" |
| **4. 좌표(Where-to)** | "지금 어디에 있고 어디로 가나?" | ③-A **Vision 2031** — 목업이 아닌 정식 섹션으로 재작성 | ↓ "누가 만드나?" |
| **5. 사람(Who)** | "누가 이걸 만드나?" | ⑤ 리더십 — 팀 페이지 데이터로 격상 + 팀 전체 링크 | ↓ "믿을 만한가?" |
| **6. 신뢰+전환** | "검증됐나? 그래서 뭘 하면 되나?" | ⑥ 파트너·인증 → ⑦ SAAI 3모드 → ⑧ CTA | 문의/제품 |

핵심 변화 2가지: **(a)** "왜 우리인가(자격)"를 뒤(④)에서 앞으로 당겨, 비전을 말하기 전에 *자격*을 먼저 보여준다. **(b)** ③의 A·B·C 3중 중첩 다크 블록을 해체해 B(방법)와 A(좌표)를 각각 독립 섹션으로 만들고, 그 사이 배경 리듬을 정리한다(§4).

> 참고: ②비전("모든 공간을, 완벽하게")과 ③-C(회사→SAAI 핸드오프), ⑦(3모드)은 모두 "회사→제품(SAAI)" 전환 장치다. 지금은 3번 반복된다. **비전은 ②에서 한 번 선언, SAAI 핸드오프는 ⑦ 하단 브릿지로 1회 통합**해 중복을 제거한다(§3.6).

## 3. 지적사항별 실행안

### 3.1 [지적 1] 리더십 — 팀 페이지 기준으로 격상

**현상(근거):** About의 리더십(⑤)은 `src/lib/leadership.ts` → `content/site/leadership.yaml`(3인: CEO 김태훈·CBO 금상호·VPE 고봉경)을 쓰고, **이니셜 원형 아바타**(`{l.initials}`)만 렌더한다. 반면 팀 페이지(`TeamView.tsx`)는 `src/data/teamMembers.ts`의 풍부한 데이터 — **일러스트 초상(`avatarUrl`)·보이스 인용·스포트라이트 다이얼로그·그룹 필터** — 를 쓴다. 즉 **리더십 데이터가 이원화**돼 있고, About 쪽이 시각적으로 빈약하다.

또한 `docs/STATUS.md`에 두 개의 미결 항목이 있다:
- §Remaining **0-11** "/company/about 리더십 잘림 — 현재 코드/데이터엔 재현 안 됨"(배포본 확인 필요)
- §후속 **"leadership 인사 확인 — 자료 CTO(이수민) vs 사이트(CBO 금상호·VP 고봉경) 정합 후 bio 보강"** *(담당: 사용자)*

**방향:** About 리더십을 팀 페이지와 **같은 시각 언어**로 통일하되, About에서는 3인 요약 + "팀 전체 보기"로 연결한다.

**실행:**
1. **데이터 SOT 단일화 검토** — `teamMembers.ts`에서 `isLeadership` 인원을 About이 재사용하도록 `leadership.ts`를 팀 데이터에서 파생하거나, 최소한 **동일 초상 자산(`avatarUrl`)을 About 카드에도 사용**한다. 이니셜 → 일러스트 초상으로 교체하는 것만으로 팀 페이지와 톤이 맞는다.
2. **카드 격상** — 이름·직책·1줄 책임(`focus`) + 2~3문장 bio 유지, 초상 추가, 호버 시 팀 스포트라이트로 연결(선택).
3. **팀 전체 링크 추가** — 리더십 섹션 하단에 "딥핑소스 사람들 전체 보기 →" (`/company/team`) CTA. 현재 About↔Team 상호 링크가 About→Team 방향으로 없다(Team→About은 있음).
4. **인사 정합**(⚠️ 사용자 결정 필요) — CBO 금상호·VPE 고봉경(사이트) vs CTO 이수민(자료)의 실제 리더십 구성 확정. 확정 전에는 현행 3인 유지. → §6 열린 결정 D1.

**검증:** About·Team 두 페이지에서 동일 인물의 이름·직책·초상이 일치. `leadership.yaml` 3인이 `teamMembers`에도 존재하는지 대조.

### 3.2 [지적 2] 디자인 · 색상 심층 검토

**팔레트 SOT(`DESIGN.md` §색상 / `globals.css`):**
- 브랜드 블루 1개 — `--primary #376AE2` (+`-dark #2453C4` `-light #5B86EA` `-lighter #E5EDFC`)
- 보조 틸 `--secondary #26A69A` (아주 절제)
- 다크 표면 `--surface-dark #0B1220` / `--surface-dark-2 #0E1730` / 섹션에서 쓰는 `slate-950`·`slate-50`
- 그림자는 항상 블루-블랙 틴트(`shadow-card`) — 중립 회색 금지
- 규칙: **"블루는 하나. 상태색은 Material 세트. 그림자는 블루-블랙 토큰."**

**문제 진단:**
1. **다크 표면이 3종 혼용** — 히어로는 `bg-surface-dark`(#0B1220), ③향후5년은 `bg-slate-950`, 목업은 다시 `bg-surface-dark`, ⑧CTA는 `bg-slate-950`. 미묘하게 다른 검정이 섞여 "같은 다크인데 안 맞는" 인상을 준다. → **다크는 `surface-dark` 토큰으로 통일**(slate-950 직접 사용 지양).
2. **파랑 남용으로 위계 소실** — primary 파랑이 eyebrow·아이콘·보더·배지·앵커·링크·CTA에 전부 쓰여, "여기가 중요하다"는 신호가 희석된다. **파랑은 (a)비전 앵커 (b)주요 CTA (c)SAAI 핸드오프 강조에만** 남기고, 나머지 장식 파랑은 슬레이트/뉴트럴로 낮춘다.
3. **표면 중첩(③)** — 다크 섹션 > 화이트 카드(`rounded-3xl bg-white`) > 다크 목업(`bg-surface-dark`). 3중 대비 반전이 "뜨는" 원인. → §3.4에서 해체.
4. **radius·여백 리듬** — `rounded-2xl`/`rounded-3xl`가 섹션마다 섞임. 카드=`2xl`, 큰 컨테이너=`3xl`로 규칙화.
5. **섹션 배경 리듬** — 현재 다크·화이트·다크·밝음·밝음·밝음·화이트·다크. 재정렬 후 **다크는 히어로와 최종 CTA 2곳으로 제한**(DESIGN_v2 §6.3 "마케팅 본문 풀-다크 지양"과 정합), 본문은 화이트↔`--layer-section-alt(#F7F9FC)` 교차.

**실행:** 위 5개를 "디자인 토큰 정리 패스"로 묶어 처리. 새 색을 만들지 않고 **기존 토큰 사용처만 정리**한다(브랜드 블루 변경 아님 → `globals.css`/`tokens.ts`/`icon.svg` 동시수정 규칙 무관).

**검증:** 다크 표면 hex 실측(모두 `surface-dark` 계열인지), 파랑 사용처를 grep해 3범주로 수렴했는지, 라이트/다크 대비 WCAG AA(`design:accessibility-review` 스킬로 감사).

### 3.3 [지적 3] 히어로 — "읽어도 무슨 말인지" + 시각적 clue

**현상(근거):** 히어로 H1은 `missionStatement = "REINVENT OFFLINE"`(영문 대문자, `WordRise` 애니메이션) → 리드 "오프라인을 / 다시 만듭니다" → 대비카드 2장(eyebrow "온라인은"/"현장은", master "측정으로 최적화됐다"/"거의 그대로였다") → 클로징 "딥핑소스는, 그 현장을 다시 만듭니다."

**문제:**
1. **은퇴한 슬로건이 첫 화면** — `docs/README.md`(라이브 SOT) 명시: *"마스터 카피 '모든 공간을, 완벽하게' — REINVENT OFFLINE/NEXTRISE는 은퇴."* 그런데 About 히어로는 은퇴 카피를 H1로 쓴다. (STATUS §163은 이를 "미션 블록+진화 서사"로 의도했다고 기록 — 즉 *유산으로서 의도적*이나, **첫 화면 H1으로 쓰기엔 강도가 과함**.)
2. **영문 대문자 H1 + 한국어 본문**의 톤 단절. 첫 3초에 "이 회사가 뭘 하는지"가 안 들어온다("무슨 말인지 모르겠다"의 정체).
3. **순수 텍스트, 시각 clue 0** — 다크 배경 위 텍스트만. "오프라인/공간"을 보여주는 이미지가 없다.
4. 변수명 오배치 — `heroMasterCompany`/`heroMasterOwner`인데 실제 내용은 "온라인 vs 현장" 대비. 코드 주석("두 마스터 카피 미러")과 실제 카피가 불일치.

**방향:** 히어로는 **① 한국어로 한 문장에 정체성 → ② 온라인/오프라인 대비를 시각으로 → ③ 은퇴 슬로건은 유산 배지로 강등**.

**실행:**
1. **H1 재작성** — "REINVENT OFFLINE"(영문)을 H1에서 내리고, 한국어 정체성 문장을 H1로. 후보(카피 라운드로 확정):
   - "온라인은 20년간 최적화됐습니다. 현장은, 거의 그대로입니다." (대비를 H1으로)
   - 또는 `companyIntro2` "딥핑소스는, 그 현장을 다시 만듭니다."를 H1급으로.
   - "REINVENT OFFLINE"은 **eyebrow/배지**로 강등(유산·에너지는 유지, 강도는 낮춤).
2. **시각적 clue 추가** — 오늘자 `HERO_SPACES_PLAN_v1.md`(홈 히어로 텍스트↔이미지 회전, `scripts/generate-hero-spaces.mjs`)의 **공간 이미지 세트를 About 히어로에도 재사용**. 매장·현장·물류·카페 등 "우리가 다시 만드는 현장"을 배경/사이드 비주얼로. 텍스트-이미지 1:1 순서 계약을 그대로 따른다.
   - 저사양 대안: 온라인(그래프/측정 UI) ↔ 오프라인(현장 사진) **좌우 대비 비주얼** 1장. "온라인은 측정됐고 현장은 안 됐다"를 그림 한 장으로.
3. **대비카드 정리** — 2장 카드는 유지하되 변수명·톤 정리(온라인=중립 슬레이트, 현장=파랑 강조로 "우리가 손대는 쪽"을 색으로 지시).

**검증:** 5초 테스트(첫 화면만 보고 "이 회사가 뭐하는 곳?" 답 가능한지), 모바일 뷰 텍스트 넘침, 이미지 LCP/`priority` 세팅.

### 3.4 [지적 4] Vision 2031 — 갑작스러움 + "뜨는" 디자인

**현상(근거):** ③향후5년 챕터 **블록 A**. 문제의 핵심은 **중복 표기와 표면 중첩**:
- 섹션 `NEXT.ko.aTitle` = **"Vision 2031 — 우리가 가려는 좌표"**
- 그 안 `VisionCoordinatesMockup`의 `eyebrow` = **"Vision 2031 — 우리가 가려는 좌표"**(동일!) + `heading` "5년 후, 우리 자리" + `lead`(섹션 `aLead`와 거의 동문)
- 즉 **같은 제목이 두 번, 리드가 두 번** 나온다 → "갑작스럽고 반복적".
- 목업에는 `MockupBadge label="예시 화면"`(Sample screen)이 붙는다 — **정식 회사 비전을 "예시 화면"으로 라벨링**하는 모순.
- 다크 섹션 > 화이트 카드 > 다크 목업 3중 중첩 → "뜨는 느낌".

**방향:** 목업(예시 화면) 프레이밍을 버리고, **Vision 2031을 정식 콘텐츠 섹션**으로 승격. 좌표 은유(익명화·공간·운영 교집합 + 2025→2031 궤적)는 살리되 단일 위계로.

**실행:**
1. **중복 제거** — 섹션 헤더(`aTitle`+`aLead`)와 목업 내부 헤더(`eyebrow`+`heading`+`lead`) 중 **하나만**. 섹션 헤더를 정본으로 두고, 목업 컴포넌트는 헤더 없는 "다이어그램 본체"만 렌더하도록 분리.
2. **"예시 화면" 배지 제거** — 이건 제품 목업 갤러리(`/demo`)용 장치다. 회사 비전에는 부적절 → `MockupBadge` 미표시.
3. **표면 중첩 해체** — 두 안 중 택1(→ D2):
   - (A) Vision 2031을 **화이트/라이트 섹션**으로 빼서 다크 카드 없이 좌표 그래픽만. (권장 — 배경 리듬도 정리됨)
   - (B) 다크 유지하되 화이트 카드를 없애고 다크 위에 직접 그래픽(중첩 1단계 축소).
4. **"갑작스러움" 완화** — §2 재배치대로 Vision 2031을 **③방법(3층) 다음**에 두면 "이렇게 일한다(방법) → 그래서 5년 뒤 여기로 간다(좌표)"로 인과가 생겨 덜 갑작스럽다.
5. **좌표 5개(01~05) 밀도** — 레일이 조밀하다. 05(회사 정체성)를 터미널로 시각 강조하는 현 구조는 좋음. 01~04는 카피 축약(각 1줄).

**검증:** 페이지에 "Vision 2031" 문자열이 **1회만** 노출되는지 grep, "예시 화면/Sample screen" 미노출, 좌표 섹션 배경 대비 단계 수 확인.

### 3.5 [지적 5] "익명화 위에 공간 지능, 그 위에 에이전트 AI" — 애매한 3층

**현상(근거):** ③블록 B. `methodHeading = "익명화 위에 공간 지능, 그 위에 에이전트 AI"` — 이 카피는 **명백히 수직 적층(stack)** 은유다(익명화가 바닥 → 그 위 공간지능 → 그 위 에이전트). 그런데:
- 실제 렌더는 `VisionDiagram` = **좌→우 파이프라인** (Physical World → Privacy Shield → Spatial AI 카드). 카피(수직)와 그림(수평)이 **불일치**.
- `VisionDiagram`의 마지막 카드는 제목 "Spatial AI" + 부제 "Agentic AI"를 **한 카드에 욱여넣음** → 3층 중 2·3층이 뭉개져 "애매".
- 바로 아래 `ProcessStepper`가 같은 3단계를 01/02/03로 **또 반복**(카피는 `methodSteps`의 term/promise). 즉 **헤딩(적층 텍스트) + VisionDiagram(수평) + ProcessStepper(번호) 3중 표현**.
- 다크 위 화이트 카드 안이라 §3.4와 같은 중첩 문제 공유.

**방향:** **하나의 깔끔한 "3층 스택 다이어그램"** 으로 통합. 카피의 "위에 / 그 위에"를 시각적으로 진짜 층으로 그린다.

**실행:**
1. **전용 컴포넌트 신설** `SpatialStackDiagram`(가칭) — 3층 수직 스택:
   - **바닥: 익명화 · Anonymizer** (SEAL, `--primary` 진하게 = 토대) — "모든 것의 시작"(brand-canon `saaiPromiseLayer.sub` "모든 것은 익명화에서 시작합니다")
   - **중간: 공간 지능 · Spatial AI** — 좌표 위에 공간을 읽음
   - **상단: 에이전트 AI · Agentic AI** — 무엇을 바꿀지 제안(결정은 사람)
   - 각 층에 `methodSteps`의 term/promise 1줄, 층 사이 "위에 쌓임" 시각(상승 화살표 또는 물리적 적층).
2. **중복 제거** — `VisionDiagram`(수평 파이프라인)과 `ProcessStepper` 중복을 스택 하나로 대체. 수평 흐름이 필요하면 스택 위에 운영 루프(Analyze→Detect→Act→Learn, `operatingLoop`)를 별도 1줄로.
3. **컴포넌트 재사용 정합** — `docs/SPATIAL_AGENTIC_DIAGRAM_WORK_ORDER_v1.md`가 `/technology/spatial-ai`·`agentic-ai`용 다이어그램(3-pillar·`PosJoinDiagram`·`LearningFlywheelDiagram`)을 정의 중. **About의 3층 스택과 기술 페이지 다이어그램이 같은 시각 언어를 쓰도록** 그 작업지시서와 컴포넌트/색/라벨을 맞춘다(익명화=토대라는 서열을 두 곳 다 동일하게).
4. **라벨 정합** — brand-canon의 카테고리/기둥 표기와 통일: 익명화(SEAL)·공간(Spatial)·에이전트(Agentic). "Spatial AI"와 "Agentic AI"를 한 카드에 합치지 않는다.

**검증:** 카피("위에/그 위에")와 그림(아래→위 적층)이 일치하는지, About 3층과 기술 페이지 다이어그램 라벨/색 대조, 모바일에서 스택이 세로로 자연스럽게 무너지는지.

### 3.6 전환(CTA) 설계 — "그래서 뭘 하면 되나"

**현상:** CTA성 장치가 흩어져 있다 — ③-C SAAI 핸드오프, ⑥ 엔터프라이즈/파트너십 링크, ⑦ 3모드+제품 전체, ⑧ 문의하기. 회사→SAAI 전환이 ②·③-C·⑦ 3번 반복.

**방향:** **1차 전환은 "문의하기", 2차는 "SAAI/제품 보기"** 로 위계를 정하고, 반복을 통합.
- **본문 중간(soft):** ③방법·④좌표 뒤 "당신의 공간부터 →" (제품/데모) 1개.
- **하단(hard):** ⑦ SAAI 3모드 브릿지 → ⑧ 최종 CTA "문의하기"(`ownerCta`가 아닌 중립 CTA — About은 점주 전용 아님, brand-canon D4 규칙).
- **아웃바운드 분기:** ⑥의 엔터프라이즈·파트너십 링크는 유지(도입/협업 문의 경로).

**검증:** 페이지당 "문의하기" 최종 CTA 1개로 수렴, 회사→SAAI 전환 노출 횟수 2회 이하.
