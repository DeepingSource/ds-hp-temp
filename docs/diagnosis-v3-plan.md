## 1. 문제 정의

v2로 대화형 연출은 갖췄지만, 실제 사용에서 두 가지 문제가 확인됐다.

### 1-A. 진단이 너무 빨리 끝난다 (질문 부족)

현재 흐름은 Q1(역할) → Q2(업종) → Q3(문제 클러스터) → [Q4 동점 해소, 조건부] → 결과.

- Q1은 진단 질문이 아니라 **라우팅 게이트**다(owner 배제·카드 분기). 사용자 입장에서 "진단다운 질문"은 Q2·Q3 두 개뿐.
- Q4(동점 해소)는 클러스터에 시나리오가 2개 이상일 때만 등장한다. 21개 시나리오 중 **1-slug 클러스터가 6개**라 상당수 경로가 **3클릭 만에 결과**로 직행한다. (convenience 3개 클러스터 전부, cafe:merchandising, mart:congestion, exhibition:congestion)
- 수집 신호가 역할·업종·문제뿐이라 결과 개인화 재료도 부족하다. 규모·증상 양상·도입 목표를 모르니 결과 패널과 라우팅 허브가 누구에게나 같은 얼굴이다.
- 진행 표시가 3단계↔4단계로 흔들려(“3 중 3”이었다가 “4 중 4”) 총량 감각도 불안정하다.

**증상**: "질문 몇 개 하지도 않았는데 답이 나와버린다" → 진단이 아니라 필터로 느껴지고, 결과의 설득력·신뢰가 깎인다.

### 1-B. 스크롤이 너무 빨라 읽을 수 없다 (페이싱)

`DiagnosisConversation.tsx`의 연출 파라미터가 문제의 원인이다.

| 현재 구현 | 문제 |
|---|---|
| `bottomRef.scrollIntoView()`를 `[step, transcript.length, isTyping, isAnalyzingResult]` 변화마다 실행 | 한 번의 답변에 3~4회 연속 스크롤 발동. 항상 **맨 아래 앵커** |
| 타이핑 인디케이터 450ms 후 질문+옵션이 동시에 팝 | 읽을 틈 없이 화면이 갈아치워짐 |
| 결과: analyzing 750ms 후 **긴 결과 패널이 통으로** 등장 + 바닥으로 스크롤 | 결과 패널은 위에서부터 읽는 콘텐츠(제목→접근→수치→허브→CTA)인데 시작점이 화면 밖으로 밀려나감. **가장 치명적** |
| 연결 멘트(`understanding`, `finalQuestion`)가 i18n에 있으나 미사용 | 대화 리듬 없이 질문만 연타 |

**원칙 위반**: 채팅 UI의 "bottom-follow"는 짧은 메시지 교환에 맞는 관성이다. 긴 결과 문서를 바닥 앵커로 밀어 넣으면 안 된다.

## 2. 목표

한 문장으로: **6문항의 진짜 진단 — 질문마다 대화의 호흡이 있고, 새 내용은 항상 "읽기 시작할 위치"에 나타난다.**

- 질문: 3(±1) → **6문항 고정** (프리셋 진입 시 5). 진행바가 흔들리지 않는다.
- 신호: 역할·업종·문제에 **규모·증상·도입 목표** 추가 → 결과 카피와 라우팅 허브 개인화.
- 페이싱: 답변→반응 멘트→타이핑→질문→칩 순차 공개. 스크롤은 **새 발화의 상단** 앵커.
- 결과: 상단 정렬 후 섹션 순차 공개. 이후 자동 스크롤 금지 — 읽는 속도는 사용자의 것.
- 로직: 규칙 기반 유지(비용 0, 정적 배포). LLM 위장 금지 원칙(v2 §4) 유지.

## 3. 새 질문 트리 — 6문항 설계

```
Q1 역할        (기존 유지 · owner → exit-owner 유지)
Q2 업종        (기존 유지 · 프리셋 진입 시 확인 칩으로 대체)
Q3 규모        (신규 · 역할에 따라 문구 분기)
Q4 문제 영역   (기존 Q3 · 클러스터 선택)
Q5 증상 확인   (기존 Q4 확장 · 전 경로 필수화)
Q6 도입 목표   (신규 · CTA/라우팅 우선순위 결정)
→ 결과
```

결과 slug는 여전히 **Q4+Q5**에서 결정된다. Q1·Q3·Q6은 결과를 바꾸지 않는 **개인화 신호**다 — 이 구분을 유지해야 CLUSTER_MAP 파생 구조(diagnosisData.ts)를 건드리지 않고 확장할 수 있다.

### Q3 규모 (신규)

역할 답변에 따라 문구만 분기, 신호 스키마는 공통(`scale: 'single' | 'small' | 'mid' | 'large'`).

| 역할 | 질문(ko 초안) | 선택지 |
|---|---|---|
| hq_sv | 관리하시는 매장이 몇 곳인가요? | 2~9곳 (small) / 10~49곳 (mid) / 50곳 이상 (large) |
| exec | 검토 범위가 어느 정도인가요? | 일부 지점 파일럿 (small) / 사업부 단위 (mid) / 전사 (large) |
| owner(계속 진행 시) | 현장 규모가 어느 정도인가요? | 소형 매장 (single) / 중대형 매장 (small) |
| logistics·exhibition 업종 | 현장 규모가 어느 정도인가요? | 단일 현장 (single) / 복수 현장 (mid) / 전국 단위 (large) |

주의: Q3는 Q2(업종) **뒤**에 두는 게 자연스러운 경우(물류·전시는 "매장 수"가 어색)가 있어, 실제 순서는 역할→업종→규모로 확정. 규모 질문 카피는 업종 슬러그를 보고 매장형/현장형 2종 중 선택.

### Q4 문제 영역 (기존 Q3 유지)

변경 없음. universal 옵션(privacy/unsure)도 유지하되, exit 시점이 6문항 중 4번째가 되므로 진행바 표기는 "4 / 6"에서 이탈로 처리.

### Q5 증상 확인 (기존 Q4 tiebreak의 전 경로 필수화)

핵심 변경. 현재 2+ slug 클러스터에만 있는 tiebreak를 **모든 클러스터**로 확장한다.

- **2+ slug 클러스터(6개)**: 기존 TIEBREAK_QUESTIONS 그대로 → 답이 결과 slug를 결정.
- **1-slug 클러스터(6개)**: 신규 "증상 확인" 질문 → 답이 slug를 바꾸지 않지만 `symptom` 신호로 저장, 결과 패널 도입부 카피에 반영(§5). 진단의 "문진" 감각을 만드는 장치이자, 결과 첫 문장을 사용자의 언어로 되돌려주는 재료.

신규 작성이 필요한 1-slug 클러스터 증상 질문(ko 초안, en/jp는 구현 시 3로케일 동시):

| 클러스터 | 질문 | 선택지 (→ symptom 신호) |
|---|---|---|
| convenience:theft_loss | 손실이 주로 언제 발생하나요? | 심야·새벽 시간대에 집중돼요 (night) / 시간대를 특정하기 어려워요 (unknown) |
| convenience:inventory | 결품을 언제 알게 되나요? | 고객이 찾을 때에야 알아요 (reactive) / 정기 점검 때 몰아서 발견해요 (batch) |
| convenience:merchandising | 진열 기준이 매장마다 지켜지고 있나요? | 본사 기준과 실제 매장이 자주 달라요 (drift) / 기준 자체를 데이터로 만들고 싶어요 (no-baseline) |
| cafe:merchandising | 좌석 회전이 안 되는 양상이 어떤가요? | 특정 좌석만 장시간 점유돼요 (hotspot) / 전체적으로 체류가 길어요 (overall) |
| mart:congestion | 계산대 혼잡이 어떤 패턴인가요? | 특정 시간대에 몰려요 (peak) / 예측이 안 되게 불규칙해요 (random) |
| exhibition:congestion | 혼잡이 어디서 생기나요? | 특정 구간·전시물 앞에 몰려요 (spot) / 입퇴장 동선 전체가 밀려요 (flow) |

### Q6 도입 목표 (신규 · 전 경로 공통)

마지막 질문. `finalQuestion` 연결 멘트("마지막 질문이에요.")를 여기서 사용.

질문(ko 초안): **"지금 어느 단계에 계세요?"**

| 선택지 | 신호 | 결과에 미치는 영향 |
|---|---|---|
| 문제가 실제 손실로 이어지고 있어 빨리 해결해야 해요 | `goal: urgent` | 주 CTA = 상담 신청 강조(현행 유지), 허브에 사례 카드 최우선 |
| 다음 분기·연간 계획에 반영하려 검토 중이에요 | `goal: planning` | 허브에 가격 시뮬레이터·엔터프라이즈 카드 우선 |
| 어떤 기술인지 먼저 이해하고 싶어요 | `goal: research` | 허브에 기술(Anonymizer)·제품 카드 우선, 부 CTA(자세히 보기)를 시각적으로 동급 처리 |

### 단계 수 규칙

- 기본 경로: 6문항 고정. `totalSteps = 6`, 조건부 증감 없음 → 진행바 안정.
- 런처 프리셋(industry)이 있으면 Q2가 확인 칩으로 대체되지만 **스텝 수는 유지**(확인도 한 스텝) — 계산 단순화, 흔들림 방지.
- exit 경로(owner/privacy/unsure)는 기존과 동일하게 중도 이탈.

## 4. 대화 카피 시나리오 — 연결 멘트 시스템

미사용 중인 `chatConnectors`를 실제로 배선하고 확장한다. 원칙: **반응 멘트는 사용자의 답을 짧게 되받는 한 문장**. 과장 금지, "분석 중" 위장 금지(Quiet Utility).

### 스키마 확장 (diagnosis-i18n.ts)

```ts
chatConnectors: {
  understanding: string;            // 기존 — 범용 수긍
  finalQuestion: string;            // 기존 — Q6 앞에서 사용
  analyzing: string;                // 기존 — 결과 직전
  rewindHover: string;              // 기존
  relatedContentHeading: string;    // 기존
  ackIndustry: (label: string) => string;   // 신규 — Q2 답 후
  ackCluster: (label: string) => string;    // 신규 — Q4 답 후
  ackScaleLarge: string;                    // 신규 — 규모 large 답 후
}
```

### 경로별 멘트 배치(ko 초안)

| 시점 | 멘트 | 비고 |
|---|---|---|
| Q2 답 후 | "{업종} 현장이시군요. 자주 듣는 문제들을 기준으로 여쭤볼게요." | ackIndustry |
| Q3 답(large) 후 | "규모가 있으시네요. 전사 관점도 함께 고려하겠습니다." | large일 때만. small/mid는 멘트 생략(모든 답에 반응하면 소음) |
| Q4 답 후 | "네, {클러스터 라벨} 쪽이군요. 조금만 더 구체적으로 볼게요." | ackCluster |
| Q5 답 후 → Q6 앞 | "마지막 질문이에요." | 기존 finalQuestion |
| Q6 답 후 | "조건에 맞는 사례와 제품을 확인하고 있어요..." | 기존 analyzing |

멘트는 **어시스턴트 말풍선 1개**로 질문 버블과 분리 렌더(transcript에는 질문만 기록, 멘트는 휘발). 3로케일 동시 작성 필수.

## 5. 신호 → 결과 반영 맵 (routing.ts · ResultPanel 확장)

수집 신호: `persona, industry, scale, cluster, symptom, goal, privacySelected`.

| 신호 | 반영 위치 | 규칙 |
|---|---|---|
| symptom | 결과 도입부 한 문장(신규) | "심야 시간대에 집중된 손실이라면, 이 사례가 가장 가깝습니다."처럼 사용자의 답을 되받아 결과와 연결. slug당 고정 카피가 아니라 symptom 키 기반 템플릿 |
| scale=large 또는 persona=exec | 허브 카드 | 엔터프라이즈 카드 승격(현행 exec 조건에 scale 조건 추가) |
| scale=small/mid + persona=hq_sv | 허브 카드 | 가격 시뮬레이터 카드 + 매장 수 프리셋 쿼리(`?stores=10`) 전달 — v2 §5의 "검토" 항목 확정 |
| goal=urgent | CTA·허브 | 사례 카드 최우선, 주 CTA 현행 강조 유지 |
| goal=planning | 허브 | 가격/엔터프라이즈 카드 우선순위 상향 |
| goal=research | CTA·허브 | 기술·제품 카드 상향, 부 CTA 시각 동급 |
| 기존 신호 | 기존 규칙 유지 | 사례>제품>가격>기술 기본 우선순위에 goal 가중치를 얹는 방식(재정렬이지 추가 아님). 카드 최대 4장 컷 유지 |

## 6. 페이싱·스크롤 재설계 사양

### 6-1. 스크롤 앵커 원칙

**"새 어시스턴트 발화의 상단"을 스크롤 컨테이너 상단 근처(offset 8~16px)에 정렬한다. 바닥 앵커 전면 폐지.**

- 구현: `bottomRef` 제거. 최신 어시스턴트 버블(또는 결과 패널 루트)에 ref → 컨테이너의 `scrollTo({ top: el.offsetTop - 12, behavior: 'smooth' })`.
- 발동 조건: **새 어시스턴트 콘텐츠가 나타날 때 1회만**. 옵션 칩 렌더·타이핑 인디케이터 표시·transcript 추가로는 발동하지 않는다(현재 4개 의존성 → 1개 이벤트로 축소).
- 사용자 답변 직후에는 스크롤하지 않는다 — 반응 멘트/다음 질문이 나타나는 시점에 한 번.
- 질문+옵션 목록이 화면보다 길면(Q2 업종 그리드) 질문 상단 정렬 덕에 자연스럽게 위에서부터 읽고 아래로 스크롤하게 된다.
- 사용자가 수동 스크롤 중(직전 500ms 내 wheel/touch)이면 자동 스크롤 스킵 — 읽기를 끊지 않는다.

### 6-2. 공개 타이밍 (질문 스텝)

```
사용자 답 선택
  → 250ms   반응 멘트 버블 (해당 시점에만, §4 표 참조)
  → 600ms   타이핑 인디케이터 (기존 450 → 600ms)
  → 0ms     질문 버블 표시 + 스크롤 앵커 1회
  → 250ms   옵션 칩 페이드인(스태거 40ms/개)
```

- 총 리듬: 답 → 다음 질문 완전 표시까지 약 1.1초. "즉답 위저드"와 "느린 챗봇" 사이.
- `prefers-reduced-motion`: 전 단계 0ms, 즉시 표시(기존 정책 유지). 스크롤도 `behavior: 'auto'`.

### 6-3. 결과 공개

```
Q6 답
  → analyzing 인디케이터 900ms (기존 750 → 900)
  → 결과 패널 마운트, 패널 상단을 컨테이너 상단에 정렬 (이후 자동 스크롤 없음)
  → 섹션 순차 공개: 진단 헤더+문제 확인(0ms) → symptom 되받기 문장(150ms)
     → 접근 3단계(350ms) → 수치(600ms) → 허브 카드(850ms) → 관련 솔루션·용어집·CTA(1050ms)
```

- 순차 공개는 opacity/translate 조합, 각 300ms 트랜지션. 레이아웃 자리는 처음부터 확보(레이아웃 시프트 금지 — 스크롤바 튐 방지).
- 결과가 화면보다 긴 것은 정상이다. **읽는 스크롤은 사용자 몫** — 유일한 개입은 최초 상단 정렬 1회.
- 모달(compact)에서는 결과 단계에서 모달 높이를 고정(예: `max-h-[85vh]` 유지)해 순차 공개로 인한 컨테이너 출렁임 방지.

### 6-4. 되감기(수정) 시

- 되감기 시 타이핑 연출 생략, 해당 질문 상단 정렬만 수행 — 이미 본 질문에 연출을 반복하지 않는다.

## 9. 리스크와 반론

- **6문항이 길어서 이탈이 늘 수 있다** → Q5·Q6은 선택지 2~3개의 가벼운 질문이고, 반응 멘트가 체감 길이를 줄인다. 그래도 Phase 4 지표로 검증하고, 이탈이 유의하게 늘면 Q3(규모)를 결과 후 질문으로 강등하는 퇴로를 남긴다.
- **증상 질문(1-slug)의 답이 결과를 안 바꾸는 건 기만 아닌가** → 답을 결과 카피에 실제로 반영(§5)하므로 위장이 아니다. 반영 없이 질문만 늘리는 것은 금지선.
- **연출 지연 총합(~1.1s/문항)이 답답할 수 있다** → reduced-motion 외에도, 되감기 재방문 시 연출 생략(§6-4)으로 반복 비용 제거. 수치는 구현 후 실기기에서 ±200ms 튜닝 여지.
